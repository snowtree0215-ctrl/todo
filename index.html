Chapter 1 · 기초편
웹 앱은 어떻게 생겼나요?
웹 앱을 만들기 전에 전체 구조를 이해해두면 AI에게 요청하고 설명하기가 훨씬 쉬워져요.
프론트엔드 (화면)꼭 알기

사용자 눈에 보이는 모든 것이에요. 버튼, 텍스트, 이미지, 입력창. HTML/CSS/JavaScript로 만들어지고 사용자의 브라우저에서 실행돼요.

식당의 홀 — 손님이 직접 보고 앉고 메뉴를 고르는 공간이에요.
백엔드 (서버)알면 좋음

사용자 눈에 안 보이지만 실제 계산·저장·로그인 처리가 일어나는 곳이에요. Python, Node.js 같은 언어로 만들고 서버 컴퓨터에서 실행돼요.

식당의 주방 — 손님은 못 들어가지만 음식이 만들어지는 곳이에요.
데이터베이스 (DB)알면 좋음

데이터를 저장하는 창고예요. 회원 정보, 게시글, 주문 내역 등이 여기에 저장돼요. 간단한 앱은 브라우저의 localStorage나 Google Sheets를 대용으로 쓰기도 해요.

식당의 냉장고 — 재료(데이터)를 보관해두고 필요할 때 꺼내요.
클라이언트 vs 서버꼭 알기

클라이언트는 요청하는 쪽(브라우저, 앱)이고, 서버는 응답하는 쪽이에요. "클라이언트 사이드 처리" = 서버 없이 브라우저에서만 실행된다는 뜻이에요.

AI가 "클라이언트에서 처리해요" 하면 = 내 브라우저에서만 동작하는 코드. 서버 비용이 안 들어요.
정적 vs 동적 웹사이트꼭 알기

정적(Static): 내용이 고정. 포트폴리오, 소개 페이지, 단순 웹앱. GitHub Pages로 무료 배포 가능.
동적(Dynamic): 로그인, 실시간 데이터, 개인화 콘텐츠. 서버가 필요해요.

처음 만드는 앱은 정적으로도 충분해요. localStorage로 데이터를 저장하면 서버 없이 개인 앱처럼 쓸 수 있어요.
Chapter 2 · 기초편
개발에서 자주 쓰는 언어들
언어 이름만 들어도 막막하죠. 각각 어떤 역할인지만 알면 충분해요.
HTML — 뼈대꼭 알기

"여기 제목", "여기 버튼", "여기 이미지" 구조를 잡아요. 꺾쇠(< >)로 감싸는 태그를 씁니다.

<!-- 주석: 이런 식으로 생겼어요 --> <h1>안녕하세요</h1> <button>클릭하기</button>
CSS — 디자인꼭 알기

색, 폰트, 크기, 여백, 버튼 모양을 정해요. HTML이 뼈대라면 CSS는 옷이에요.

button { background-color: blue; border-radius: 8px; }
JavaScript (JS) — 동작꼭 알기

버튼 클릭 반응, 데이터 계산, 서버 통신이 JS로 이루어져요. 세 언어 중 가장 복잡하지만 AI가 전부 써줘요.

"JS", "자바스크립트", "스크립트"는 모두 같은 말이에요. Java와는 완전히 다른 언어예요 — 이름만 비슷해요!
Java vs JavaScript완전히 다름

이름만 비슷하고 완전히 다른 언어예요. Java는 주로 기업용 백엔드나 안드로이드 앱에, JavaScript는 웹 프론트엔드에 써요.

웹 개발 맥락에서 "스크립트", "JS" = JavaScript예요. Java를 쓸 일은 거의 없어요.
TypeScript (TS)나중에

JavaScript에 타입 체크 기능을 추가한 언어예요. AI가 TS로 코드를 쓰기도 해요. "JS랑 거의 같은데 더 엄격한 것"으로 이해하면 충분해요.

AI가 .ts 파일을 만들었다면 TypeScript예요. 당황하지 말아요 — JS처럼 쓰면 돼요.
Chapter 3 · 실전편
AI가 준 코드, 어떻게 읽나요?
직접 짤 줄 몰라도 구조를 눈으로 읽을 수 있으면 대화가 훨씬 쉬워져요.
함수 (function)핵심 개념

어떤 작업을 묶어서 이름을 붙인 것이에요. ()가 붙어 있으면 함수 호출이에요.

function greet(name) { return "안녕, " + name; } greet("홍길동"); // → "안녕, 홍길동"
레시피 카드. 이름을 붙여두고 필요할 때마다 꺼내 써요.
변수 (variable)핵심 개념

값을 담아두는 이름표예요. const는 변경 불가, let은 변경 가능.

const appName = "내 앱"; // 안 바뀜 let count = 0; // 바뀔 수 있음 count = count + 1; // 이제 1
조건문 (if)핵심 개념

"만약 ~이면 이렇게"를 표현해요. 거의 모든 코드에 등장해요.

if (isLoggedIn) { showDashboard(); } else { showLoginPage(); }
화살표 함수 (Arrow Function)자주 보임

함수를 짧게 쓰는 문법이에요. AI가 자주 씁니다.

// 일반 함수와 같은 동작이에요 const add = (a, b) => a + b; items.map(item => item.name)
=> 가 보이면 "아, 함수구나"라고 인식하면 돼요.
반환(return) vs 출력(console.log)코드 읽기 팁

return: 함수가 값을 돌려주는 것. 다른 코드에서 그 값을 쓸 수 있어요.
console.log: 개발자 콘솔에 확인용으로 출력하는 것. 실제 앱 화면과는 무관해요.

AI가 console.log를 많이 쓴다면 디버깅용이에요. 배포 전에 지워도 되는 코드예요.
Chapter 4 · 실전편
에러 메시지 해석하기
빨간 글씨가 떠도 당황하지 말아요. 패턴만 알면 겁이 훨씬 덜 나요.
에러 메시지 구조핵심 패턴
에러 종류: 뭐가 잘못됐는지 (예: TypeError)
에러 내용: 구체적으로 어떤 문제인지
위치: 몇 번째 줄에서 났는지 (at line 23)
전부 이해 안 해도 돼요. 에러 메시지 전체를 복사해서 AI에게 주면 돼요.
자주 보는 에러 종류참고용

is not defined → 없는 변수·함수를 썼어요
Cannot read properties of undefined → 값이 없는데 뭔가 꺼내려 했어요
SyntaxError → 괄호가 안 닫혔거나 오타
404 Not Found → 요청한 주소·파일이 없어요
401 Unauthorized → 로그인 필요 or 권한 없음
500 Internal Server Error → 서버 쪽 문제

콘솔(Console) 여는 법꼭 알기

Windows/Linux: F12 → Console 탭
Mac: Cmd + Option + J

앱이 이상하면 콘솔부터 열어요. 빨간 글씨 = 에러, 노란 글씨 = 경고예요.
"빌드 실패" vs "런타임 에러"구분 팁

빌드 에러: 앱을 시작하기 전에 나는 에러. 터미널에 떠요.
런타임 에러: 앱이 실행 중에 나는 에러. 콘솔에 떠요.

Chapter 5 · 실전편
파일 구조와 배포
프로젝트 파일 구조를 알면 AI가 뭘 만들고 있는지 훨씬 쉽게 파악돼요.
기본 웹앱 파일 구조꼭 알기
my-app/ ├── index.html # 시작 페이지 ├── style.css # 디자인 ├── script.js # 기능 └── images/ # 이미지 폴더
단일 HTML 파일 앱은 CSS·JS를 HTML 안에 전부 넣기도 해요. 간단한 앱에 많이 쓰이는 방식이에요.
로컬 vs 배포꼭 알기

로컬(local): 내 컴퓨터에서만 돌아가요. 주소가 localhost:3000처럼 생겼어요.

배포(deploy): 인터넷 서버에 올려서 누구나 접속 가능. 실제 URL이 생겨요.

"로컬에서는 되는데 배포 후에 안 돼요"는 매우 흔한 상황이에요. 환경이 달라서 생기는 문제예요.
무료 배포 도구꼭 알기

GitHub Pages: HTML 파일 무료 배포. 정적 사이트에 딱이에요.
Vercel / Netlify: React 같은 복잡한 앱도 무료 배포 가능.
Replit: 코딩 + 배포를 한 곳에서.

Git과 GitHub알면 좋음

Git: 코드 변경 내역을 저장하는 도구. 버전 관리.
GitHub: Git 코드를 인터넷에 올려두는 곳.

Git = 저장 버튼. GitHub = 클라우드. 언제든 이전 상태로 되돌릴 수 있어요.
환경변수 / API Key 보안알면 좋음

API 키 같은 비밀 정보는 코드에 직접 넣으면 안 돼요. .env 파일에 따로 보관해요.

API 키를 코드에 넣고 GitHub에 올리면 누구나 볼 수 있어요. 키가 탈취당할 수 있어요.
Chapter 6 · 실전편
데이터가 어떻게 움직이나요?
"데이터를 어디서 가져오고, 어디에 저장하지?" 이 흐름을 알면 앱 전체 구조가 보여요.
localStorage — 서버 없이 데이터 저장핵심 개념

서버 없이 브라우저 자체에 데이터를 저장해요. 탭을 닫아도 데이터가 유지돼요. 단, 같은 기기·브라우저에서만 보여요.

localStorage.setItem('todos', JSON.stringify(list)); // 저장 const saved = JSON.parse(localStorage.getItem('todos')); // 불러오기 localStorage.removeItem('todos'); // 삭제
간단한 할일 앱, 설정 저장, 메모 앱은 localStorage만으로 충분해요. 서버 없이 "앱처럼" 동작해요.
fetch / API 호출핵심 개념

외부 서버에 데이터를 요청하는 행위예요.

const response = await fetch('https://api.example.com'); const data = await response.json();
요청(Request): "이 데이터 줘"
응답(Response): 서버가 데이터를 돌려줌
JSON: 주고받는 데이터 형식
JSON 형식꼭 알기
{ "name": "홍길동", "skills": ["기획", "디자인"] } // data.name → "홍길동" // data.skills[0] → "기획"
{ }는 객체(하나의 덩어리), [ ]는 배열(목록). 이 두 가지가 JSON의 전부예요.
동기 vs 비동기 / async·await개념 정리

API 요청은 시간이 걸려서 비동기로 처리해요. await을 붙이면 "이 작업 끝날 때까지 기다려"예요.

카페 진동벨. 커피 나올 때까지 기다리는 대신 자리에 앉아 다른 일 하다가 벨 울리면 가져오는 방식이에요.
데이터 저장 방식 비교알면 유용
localStorage: 브라우저에 저장. 서버 불필요. 같은 기기에서만 접근.
Firebase / Supabase: 클라우드 DB. 여러 기기에서 동기화. 로그인 연동 가능.
Google Sheets: 스프레드시트를 DB처럼 활용. 간단한 앱에 적합.
Chapter 7 · 실전편
AI에게 잘 요청하는 법
같은 요청도 어떻게 하느냐에 따라 결과가 크게 달라져요. 이 패턴들을 익히면 개발 속도가 눈에 띄게 빨라져요.
AI는 대화를 닫으면 기억을 잃어요반드시 알기

AI는 대화창을 닫으면 이전 내용을 전혀 기억하지 못해요. 새 대화를 시작할 때는 항상 배경 설명부터 해줘야 해요.

나쁜 예: "그거 고쳐줘" (뭐가 뭔지 모름)

좋은 예: "HTML+JS로 만든 할일 앱이야. 완료 버튼 누르면 취소선이 생겨야 하는데 아무 반응이 없어. 코드 보여줄게: [코드]"
새 대화 시작 시 항상 포함할 것: ① 앱이 어떤 앱인지 ② 어떤 기술로 만들었는지 ③ 지금 어떤 문제인지
에러는 메시지 전체를 그대로 주기핵심 습관
나쁜 예: "에러 났어. 뭔가 undefined 어쩌구 하던데"

좋은 예: "이 에러 뜨는데:
TypeError: Cannot read properties of undefined (reading 'map')
at App.js:23:15"
에러 메시지 전체를 복사해서 AI에게 붙여넣으세요. 직접 해석하려고 하면 오히려 더 오래 걸려요.
원하는 결과를 구체적으로 묘사하기핵심 습관
모호한 요청: "예쁘게 만들어줘"

좋은 요청: "배경은 어두운 남색, 카드는 흰색으로. 애플 홈페이지처럼 여백이 넉넉한 스타일"

또는: "노션처럼 심플하게, 장식 없이 텍스트 중심"
레퍼런스 사이트를 언급하는 게 가장 빠릅니다. "XXX 느낌으로", "YYY 스타일처럼"이 추상적 형용사보다 훨씬 정확해요.
한 번에 하나씩, 단계적으로 만들기핵심 습관
한 기능씩 → 직접 확인 → 다음 기능 순서가 가장 안정적이에요.

추천 흐름:
1단계: "기본 레이아웃만 먼저 만들어줘" → 브라우저에서 확인
2단계: "할일 추가 기능 넣어줘" → 확인
3단계: "삭제 기능 추가해줘" → 확인
한 번에 너무 많은 걸 요청하면 AI가 중간에 실수해도 어디서 틀렸는지 찾기 어려워요.
이해 안 될 때 바로 물어보기소통 팁

"방금 말한 미들웨어가 뭐야?" — 모르는 용어 즉시 질문
"버튼 눌렀을 때 그 상자 같은 게 나오는 것 있잖아" — 정확한 이름 몰라도 OK
"더 쉽게 설명해줘" / "비유로 설명해줘" / "한 단계씩 설명해줘" — 이해도에 맞게 조절 요청

AI에게 "비전공자도 이해할 수 있게"라고 요청하면 설명 방식이 완전히 달라져요.
좋은 요청 템플릿복사해서 써요
기능 추가 요청:
"HTML+JS로 만든 [앱 종류] 앱이야. 지금 [현재 상태]인데, [원하는 기능]을 추가해줘. [특이사항 있으면]."

버그 수정 요청:
"[어떤 동작을 했을 때] [어떤 문제가 생겨]. 에러 메시지: [전체 복사]. 관련 코드: [붙여넣기]"

디자인 변경 요청:
"[어떤 요소를] [어떻게] 바꿔줘. [레퍼런스 사이트]처럼 [핵심 느낌]"
Chapter 8 · 심화편
구조 관련 용어 심화
React를 쓰거나 AI가 설명할 때 자주 나오는 구조 단어들이에요.
컴포넌트 (Component)핵심 개념

화면을 이루는 독립적인 조각이에요. 헤더, 버튼, 카드 각각이 컴포넌트예요. 한 번 만들면 여러 곳에서 재사용 가능해요.

레고 블록. 블록 하나하나를 조립해서 전체 앱을 만들어요.
상태 (State)핵심 개념

앱이 지금 기억하는 값이에요. 상태가 바뀌면 화면이 자동으로 다시 그려져요(리렌더링).

신호등 색. 빨강→초록으로 바뀌면(상태 변화) 자동차가 반응(화면 변화)해요.
props (프롭스)React 용어

부모 컴포넌트가 자식에게 전달하는 설정값이에요.

<Button color="blue" text="확인" />
DOM알면 좋음

브라우저가 HTML을 읽어서 만든 구조예요. JS는 DOM을 통해 화면 요소를 읽고 바꿔요.
"DOM을 직접 조작한다" = JS로 HTML 요소를 직접 건드린다는 뜻이에요.

라이브러리 vs 프레임워크개념 정리

라이브러리: 필요한 도구를 가져다 쓰는 것. 내가 주도권을 가져요. (React, Chart.js)
프레임워크: 틀이 정해져 있고 그 안에서 코드를 채워요. (Next.js, Django)

라이브러리 = 마트에서 재료 골라 요리. 프레임워크 = 밀키트, 순서 정해짐.
헷갈리기 쉬운 용어들자주 혼동

서버 vs 서버리스: 서버리스는 서버가 없는 게 아니라, 내가 직접 서버를 관리 안 해도 되는 방식. Vercel이 대표적이에요.

실행 / 빌드 / 배포: 실행(run) = 지금 켜는 것 · 빌드(build) = 배포용 파일로 변환 · 배포(deploy) = 인터넷에 올리는 것

Chapter 9 · 심화편
앱 종류와 기술 스택
어떤 앱을 만들 건지 정확히 표현하면 AI가 훨씬 적합한 방향을 제안할 수 있어요.
기술 스택 (Tech Stack)필수 용어

앱을 만드는 데 쓰인 기술들의 조합이에요.

간단한 웹앱: HTML + CSS + JS 단일 파일
React 앱: React + Tailwind CSS + Vercel
풀스택: Next.js + PostgreSQL + Vercel
웹앱 vs 모바일앱 vs PWA종류 구분

웹앱: 브라우저. 설치 불필요. 진입 장벽 가장 낮음.
PWA (Progressive Web App): 웹앱을 홈 화면에 설치해서 앱처럼 쓰는 방식. manifest.json 추가만으로 가능.
모바일앱: 스마트폰 설치. React Native, Flutter.
데스크톱앱: PC 설치. Electron으로 웹 기술로도 가능.

처음 만드는 앱은 웹앱부터 시작하세요. PWA 설정만 추가하면 홈 화면에 앱 아이콘도 만들 수 있어요.
PWA — 웹앱을 앱처럼알면 유용

HTML 파일에 manifest.json을 추가하면 브라우저에서 "홈 화면에 추가"가 가능해요. 설치 후에는 일반 앱처럼 실행돼요.

PWA 구성 요소:
manifest.json — 앱 이름, 아이콘, 색상 정의
Service Worker — 오프라인 캐시 (선택적)
npm / 패키지알면 좋음

npm install 패키지명 = "이 도구 설치해줘". npm은 JS 도구들의 앱스토어 같은 곳이에요.

package.json은 "이 프로젝트에 필요한 도구 목록"이에요. npm install 하면 이걸 보고 자동 설치해요.
Chapter 10 · 심화편
개발 흐름과 필수 도구들
AI와 실제 개발할 때 흐름이 어떻게 돌아가는지 알아두면 덜 헷갈려요.
개발 → 테스트 → 배포 사이클전체 흐름
1. 로컬 개발: 내 컴퓨터에서 코드 작성 + 브라우저로 확인
2. 테스트: 기능이 제대로 동작하는지 직접 눌러보기
3. 배포: GitHub → Vercel 등으로 인터넷에 공개
4. 유지보수: 버그 수정, 기능 추가 → 1번으로
터미널 기본 명령어알면 유용
ls # 현재 폴더 파일 목록 cd 폴더명 # 폴더 이동 pwd # 지금 내 위치 npm install # 패키지 설치 npm run dev # 개발 서버 시작
AI 코딩 도구가 대부분 대신 실행해줘요. 명령어를 외울 필요는 없고 "이런 게 있구나" 정도로만 알아도 충분해요.
리팩토링 / 빌드 / 의존성자주 나오는 단어

리팩토링(Refactoring): 기능은 그대로 두고 코드를 더 깔끔하게 정리하는 작업. AI가 "리팩토링 해드릴까요?" = 기능 변경 없이 코드 정리.

빌드(Build): 개발용 코드를 배포용 최적화 파일로 변환하는 과정. Vercel은 자동으로 해줘요.

의존성(Dependency): 이 프로젝트가 동작하기 위해 필요한 외부 라이브러리들. package.json의 dependencies 항목에 적혀 있어요.

Chapter 11 · 심화편
AI 코딩 도구 활용법
채팅으로 개념을 물어보는 것과, 실제 코드를 짜고 파일을 만드는 것은 다른 도구예요. 둘을 구분해서 쓰면 훨씬 효율적이에요.
AI 채팅 vs AI 코딩 에이전트반드시 구분하기
AI 채팅 (ChatGPT, Claude, Gemini 등)
설명·기획·코드 검토·개념 이해·막힐 때 상담. 텍스트로 대화하는 것.

AI 코딩 에이전트 (Cursor, Windsurf, Claude Code 등)
실제 파일 생성·수정, 터미널 명령 실행, 앱 실행, 패키지 설치. 코드를 직접 작성하고 실행하는 것.
기획하고 방향 잡을 때 → AI 채팅. 실제 코드 짜고 파일 만들 때 → AI 코딩 에이전트. 이 두 가지를 번갈아 쓰는 게 효율적이에요.
AI 코딩 에이전트에게 잘 요청하는 법핵심 습관
기능 추가: "현재 앱에 검색 기능 추가해줘. 입력창에 치면 목록이 필터링되게"

버그 수정: "버튼 클릭 시 페이지가 새로고침돼. 막아줘"

전체 제작: "사용자가 메모 작성·저장할 수 있는 웹앱. localStorage 사용. 단일 HTML 파일로"
AI 코딩 에이전트에게는 결과물을 구체적으로 말하세요. "예쁘게"보다 "노션처럼 심플하게"가 훨씬 나아요.
파일을 여러 개 만들었을 때상황 대처

AI 코딩 에이전트가 한 번에 파일을 여러 개 만들고 수정할 때, 어떤 파일이 무슨 역할인지 물어봐도 돼요.

"방금 만든 파일들 각각 뭐 하는 건지 설명해줘" — 이렇게 물어보면 각 파일의 역할을 알 수 있어요.
프로젝트 지시 파일 (CLAUDE.md, .cursorrules 등)알면 유용

AI 코딩 에이전트는 프로젝트 루트에 있는 특정 파일을 세션 시작 시 자동으로 읽어요. 여기에 프로젝트 설명을 써두면 매번 설명하지 않아도 돼요.

지시 파일에 써두면 좋은 것:
- 이 앱이 어떤 앱인지 (목적, 대상 사용자)
- 기술 스택 (React? 순수 HTML? Firebase?)
- 코딩 스타일 규칙 (한국어 주석, 탭 크기 등)
- 하면 안 되는 것 (예: 외부 CSS 프레임워크 쓰지 말기)
- 자주 쓰는 명령어
도구마다 파일 이름이 달라요 — Claude Code는 CLAUDE.md, Cursor는 .cursorrules. 어떤 파일인지는 각 도구 문서에서 확인하세요.
에이전트 (Agent) — AI가 스스로 판단하고 행동하는 것개념 이해

일반 채팅에서 AI는 텍스트 답변만 해요. 하지만 코딩 에이전트 모드에서는 파일을 읽고, 코드를 수정하고, 터미널 명령을 실행하고, 결과를 확인하면서 작업을 스스로 완성해요.

AI가 "파일 수정 중...", "터미널 실행 중..." 같은 메시지를 보여주면 에이전트로 동작하고 있다는 뜻이에요.

채팅 모드: 사람이 코드 복사 → 붙여넣기 → 실행
에이전트 모드: AI가 파일 직접 수정 + 실행까지 자동
에이전트가 뭔가 잘못된 것 같으면 언제든 중간에 멈출 수 있어요. 중요한 작업 전에 Git commit으로 백업해두는 게 안전해요.
Chapter 12 · Git 실전
브랜치 · 커밋 · PR · merge
"왜 갑자기 브랜치가 생겼지?", "PR이 뭐야?", "merge는 어떻게 해?" — 이 챕터 하나로 정리해요.
commit이란?꼭 알기

변경 사항을 저장 + 메모하는 것이에요. 게임 세이브 포인트처럼 커밋 시점으로 언제든 돌아갈 수 있어요.

git add . # 변경된 파일 선택 git commit -m "로그인 기능 추가" # 저장 + 메모
push vs pull꼭 알기

push: 내 컴퓨터(로컬)의 코드를 GitHub(원격)에 올리는 것
pull: GitHub(원격)의 코드를 내 컴퓨터(로컬)로 내려받는 것

git push origin main # 로컬 → GitHub git pull origin main # GitHub → 로컬
push = 구글 드라이브에 업로드. pull = 구글 드라이브에서 다운로드.
브랜치 (Branch)란?핵심 개념

코드의 복사본 작업 공간이에요. 원본(main)을 건드리지 않고 새 기능을 시험해볼 수 있어요.

문서 편집할 때 "원본은 두고 사본에서 수정해봐" 하는 것과 같아요.
# 새 브랜치 만들고 이동 git checkout -b feature/login # 이제 이 브랜치에서 작업 → main은 그대로
merge와 PR핵심 개념

merge: 브랜치에서 작업한 내용을 다른 브랜치(보통 main)에 합치는 것.

PR (Pull Request): "내 브랜치 내용을 main에 합쳐도 될까요?"라고 GitHub에서 요청하는 것. diff를 확인하고 merge 버튼을 누르면 완료.

PR 흐름:
1. 브랜치에서 작업 완료
2. GitHub에서 PR 생성
3. diff 확인 (무엇이 바뀌었는지 검토)
4. 문제 없으면 merge 버튼 클릭
diff와 머지 충돌알면 좋음

diff: 변경 사항의 차이를 보여주는 것. 어떤 줄이 추가됐는지(초록), 삭제됐는지(빨강).

머지 충돌(Merge Conflict): 두 브랜치가 같은 파일의 같은 줄을 다르게 수정했을 때 발생. Git이 "어떤 걸 쓸지 나는 모르겠으니 직접 골라"라고 멈추는 상황.

충돌이 나면 파일을 직접 열어서 원하는 내용만 남기고 나머지 표시를 지운 뒤, 다시 커밋해야 해요.
전체 흐름 한눈에 보기정리
혼자 개발하는 가장 간단한 흐름:
코드 수정 → git add . → git commit -m "메모" → git push → 배포 자동 업데이트

브랜치를 쓰는 안전한 흐름:
새 브랜치 생성 → 코드 수정 → commit → push → PR 생성 → diff 확인 → merge → main 배포
Chapter 13 · Git 실전
브랜치 실전 생존 가이드
브랜치 때문에 작업 내용이 유실되거나, 어느 브랜치에 있는지 몰라 혼란스러웠던 경험 — 이 챕터 하나로 정리해요.
지금 내가 어느 브랜치에 있는지 항상 확인하기가장 중요

작업 시작 전에 현재 브랜치를 확인하는 습관이 가장 중요해요.

git branch # 브랜치 목록 + 현재 위치(* 표시) git status # 현재 브랜치 + 변경된 파일 목록
작업 내용이 사라진 것처럼 보일 때유실 사고 원인 1

브랜치를 이동하면 파일 내용이 그 브랜치 상태로 바뀌어요. 사라진 게 아니라 다른 브랜치에 있는 거예요.

# 내가 작업하던 브랜치로 다시 이동 git checkout feature/내가-작업하던-브랜치
커밋 안 하고 브랜치 이동하면 생기는 일유실 사고 원인 2
안전한 브랜치 이동 순서:
1. 작업 완료
2. git add . → git commit -m "작업 내용" (저장 먼저)
3. 그 다음 git checkout 다른브랜치
커밋하지 않고 브랜치를 이동하는 건 위험해요. 이동하기 전에 항상 커밋하거나, stash를 쓰세요.
stash — 임시 저장유용한 도구

커밋하기엔 애매한데 일단 작업 내용을 잠깐 치워둘 때 쓰는 임시 보관함이에요.

git stash # 현재 변경사항 임시 저장 git stash pop # 임시 저장한 내용 다시 꺼내기
서랍에 잠깐 넣어두기. 나중에 꺼내면 그대로 있어요.
reset — 가장 위험한 명령어위험
git reset --soft: 커밋만 취소, 파일 변경사항은 남아있음 (비교적 안전)

git reset --mixed: 커밋 취소 + add도 취소, 파일은 남아있음

git reset --hard: 커밋 취소 + 파일 변경사항까지 전부 삭제 (위험!)
AI 도구가 reset을 하려고 할 때 어떤 종류인지, 어떤 커밋으로 돌아가는지 꼭 확인해요.
HEAD와 origin알면 좋음

HEAD: 지금 내가 작업하고 있는 현재 위치예요.
origin: GitHub에 있는 내 레포지토리(원격 저장소)의 별명이에요.

git push origin main # origin = GitHub 주소의 별명 git pull origin main # GitHub에서 내려받기
Chapter 14 · 개발 환경
작업하다 갑자기 튀어나오는 개념들
실제로 개발하다 보면 "이게 뭐지?" 싶은 단어들이 갑자기 등장해요. 자주 만나는 것들만 골라서 정리했어요.
포트 (Port) — localhost:3000의 3000자주 보임

같은 컴퓨터에서 여러 서버를 동시에 실행하기 위한 번호예요.

localhost:3000 → 3000번 포트에서 실행 중인 앱
localhost:8080 → 8080번 포트에서 실행 중인 앱
"포트 3000이 이미 사용 중" 에러 = 이미 그 번호로 다른 앱이 실행되고 있다는 뜻이에요.
환경 (Environment) — dev / staging / production꼭 알기
development (dev): 개발 중인 환경. 내 컴퓨터. 막 고쳐도 됨.
staging: 배포 전 테스트 환경. 실제랑 똑같이 만들어둔 테스트 공간.
production (prod): 실제 사용자가 쓰는 환경. 신중하게 다뤄야 함.
CORS 에러자주 만남

브라우저가 다른 주소의 서버에 요청할 때 막히는 보안 정책이에요.

CORS 에러 나면 에러 메시지 전체를 AI에게 붙여넣으면 해결해줘요.
캐시 (Cache)알면 유용

속도를 위해 데이터를 미리 저장해두는 것이에요. "수정했는데 변경이 안 보여요" 현상의 원인이 되기도 해요.

브라우저 캐시 지우기: Ctrl+Shift+R (강력 새로고침).
환경변수 / API 키 / Secrets꼭 구분하기

API 키: 서비스에 접근하는 영구적인 비밀번호 같은 것.
토큰(Token): 로그인 후 임시로 발급되는 출입증. 시간이 지나면 만료돼요.

Secrets: 배포 환경(GitHub/Vercel)에서 쓰는 비밀 설정.

정적 웹앱 (Static Web App)꼭 알기

HTML·CSS·JS 파일만으로 구성되고, 브라우저가 직접 읽어서 실행해요. 서버가 필요 없어요.

프로젝트/ ├── index.html ├── style.css └── script.js 실행 방법: index.html 더블클릭 → Chrome이 열림 → 웹앱 실행 ✅
어떤 기능이 필요할 때 서버가 생기나요?핵심 판단 기준

HTML 방식으로 충분한 경우:
계산기, 간단한 폼, 단순 정보 표시, localStorage 저장, Firebase 직접 연동

서버(Node.js·Python 등)가 필요해지는 경우:
회원가입·로그인, 여러 사람이 공유하는 DB, 파일 업로드, 이메일 발송, 결제 처리, API 키를 숨겨야 할 때

정적 웹앱 = 자판기 (전기만 꽂으면 됨) 🏧 · 서버 기반 웹앱 = 카페 (바리스타·기계·재료 전부 필요) ☕
서버 기반 기술 스택의 흐름
간단한 앱: HTML + CSS + JS → index.html 규모 있는 앱: React (화면 담당) ↕ Node.js + Express (서버 담당) ↕ SQLite / PostgreSQL (데이터 저장) → npm run dev로 실행
CI/CD알면 좋음

Continuous Integration / Continuous Deployment. 코드 올리면 자동으로 테스트하고 배포하는 파이프라인. GitHub Actions 같은 것.

GitHub Pages를 쓰면 main에 push할 때마다 자동 배포되는데, 이게 간단한 CD예요.

퀴즈
개념 퀴즈
틀려도 괜찮아요. 해설을 읽으면서 이해하면 돼요.

Q1. "클라이언트 사이드에서 처리한다"는 말의 뜻은?

클라이언트 = 브라우저(사용자 기기)예요. 서버 없이 브라우저에서만 실행된다는 뜻이에요.

Q2. const와 let의 차이는?

const(상수)는 한 번 정하면 변경 불가. let은 나중에 값을 바꿀 수 있어요.

Q3. 앱에서 빨간 에러가 떴어요. 가장 먼저 해야 할 행동은?

콘솔(F12)에서 에러 전체를 복사해서 AI에게 주는 게 가장 빠른 해결법이에요.

Q4. Java와 JavaScript의 관계는?

Java와 JavaScript는 완전히 다른 언어예요. 이름이 비슷한 건 마케팅 이유 때문이에요.

Q5. "상태(State)가 바뀌었다"는 말의 의미는?

상태(state)는 앱이 현재 기억하는 값이에요. 상태가 바뀌면 React 등은 자동으로 화면을 다시 그려줘요.

Q6. "빌드(build)"란 무엇인가요?

빌드는 개발용 코드를 배포 가능한 형태로 변환하는 과정이에요. 실행(내 컴퓨터에서 켜기)과 배포(인터넷에 올리기) 사이에 있어요.

Q7. "리팩토링(Refactoring)을 해드릴까요?"라는 AI 제안의 의미는?

리팩토링은 기능 변경 없이 코드 구조를 개선하는 작업이에요. 사용자 입장에서는 아무것도 안 바뀌어요.

Q8. localStorage에 데이터를 저장했을 때의 특징은?

localStorage는 브라우저에 영구 저장돼요. 탭을 닫거나 컴퓨터를 껐다 켜도 유지되지만, 다른 기기에서는 볼 수 없어요.

Q9. git push와 git pull의 차이는?

push = 내 컴퓨터에서 GitHub로 올리기. pull = GitHub에서 내 컴퓨터로 내려받기. 구글 드라이브의 업로드/다운로드와 같아요.

Q10. 두 브랜치가 같은 파일의 같은 줄을 다르게 수정했을 때 merge하면?

머지 충돌이 나면 파일 안에 <<<<<<< HEAD / ======= / >>>>>>> 표시가 생기고, 직접 원하는 내용을 골라서 저장해야 해요.

⚙️ 코드 기본 개념
스코프 / Scope

변수가 살아있는 범위. 함수 안에서 만든 변수는 함수 밖에서 쓸 수 없음. 이걸 모르면 "변수가 왜 없다고 하지?" 버그가 생김.

function hello() { const name = '홍길동'; // 함수 스코프 안 } console.log(name); // ❌ 오류 — 밖에서 접근 불가
콜백 / Callback

나중에 호출되도록 함수에 넘겨주는 함수. "이거 끝나면 이 함수 불러줘"의 개념.

// "파일 읽기 끝나면 이 함수(콜백)를 실행해줘" readFile('data.txt', function(result) { console.log(result); // 이게 콜백 });
비동기 / Async · Await

기다리지 않고 다음 코드 실행. 서버 응답 기다리는 동안 화면이 멈추지 않는 이유. await은 "여기서만 기다려".

// await 없이 → 데이터 오기 전에 실행됨 (버그) // await 있음 → 데이터 올 때까지 기다렸다가 실행 const data = await fetch('/api/items');
사이드 이펙트 / Side Effect

함수가 반환값 외에 외부에 영향을 주는 것. localStorage 저장, 화면 업데이트, 콘솔 출력 등.

AI가 "이 함수에 사이드 이펙트가 있어요"라고 하면: 이 함수가 뭔가를 바꾼다는 뜻.
리그레션 / Regression

이전에 잘 되던 기능이 수정 후에 망가지는 것. 버그를 고쳤더니 다른 버그가 생기는 상황.

AI가 "리그레션이 없는지 테스트해보세요"라고 하면: 기존에 잘 되던 것도 확인하세요 라는 뜻.
하드코딩 / Hardcoding

값을 변수로 빼지 않고 코드 안에 직접 박아넣는 것. 나중에 바꿔야 할 때 힘들어짐.

예: 색상값이나 URL을 코드 여기저기 직접 써넣는 것. 변수나 상수로 빼는 게 좋은 습관.
컨벤션 / Convention

팀이나 언어에서 암묵적으로 정한 코딩 규칙. 강제는 아니지만 안 따르면 이상하게 보임.

예: 변수명은 camelCase, 상수는 UPPER_SNAKE_CASE, 파일명은 kebab-case 등.
기술 부채 / Tech Debt

빠르게 개발하려고 대충 짠 코드. 지금은 잘 돌아가지만 나중에 반드시 고쳐야 하는 것. 금전적 부채처럼 이자가 붙음.

롤백 / Rollback

배포 후 문제가 생겨서 이전 버전으로 되돌리는 것. git revert나 이전 버전 재배포로 실행.

🗄️ 자료구조
스택 / Stack

마지막에 넣은 게 먼저 나오는 구조 (LIFO). 브라우저 뒤로가기, Ctrl+Z 실행취소가 이 방식.

큐 / Queue

먼저 넣은 게 먼저 나오는 구조 (FIFO). 줄 서기랑 같음. 네트워크 요청 대기열, 알림 발송 순서에 사용.

해시맵 / HashMap (객체)

key-value 쌍의 저장 구조. JavaScript의 객체(Object)가 해시맵이에요. key로 값을 바로 꺼낼 수 있어 빠름.

const map = { 'apple': 1200, 'banana': 800 }; console.log(map['apple']); // → 1200
🌐 웹과 API
엔드포인트 / Endpoint

API에서 특정 기능을 담당하는 URL 주소. "이 주소로 요청 보내면 이 기능을 해줌".

GET /api/users ← 유저 목록 POST /api/users ← 유저 추가 DELETE /api/users/123 ← 특정 유저 삭제
페이로드 / Payload

요청이나 응답에 실제로 담기는 데이터. 택배로 치면 박스(HTTP 요청) 안에 들어있는 내용물(payload).

웹훅 / Webhook

이벤트가 발생하면 자동으로 특정 URL을 호출해주는 것. 내가 계속 물어보는 게 아니라 상대방이 알려줌.

예: GitHub에서 코드 push되면 → 자동으로 배포 서버에 알림 전송
미들웨어 / Middleware

요청과 응답 사이에 끼어서 처리하는 함수. 로그인 확인, 로그 기록, 데이터 검증 등을 요청마다 공통으로 처리할 때 씀.

공항 보안 검색대처럼, 요청이 목적지에 도달하기 전에 거쳐야 하는 관문들.
멱등성 / Idempotent

같은 요청을 여러 번 해도 결과가 같은 성질. GET, PUT은 멱등, POST는 아님 (누를 때마다 새로 생성됨).

결제 버튼을 실수로 두 번 눌러도 한 번만 결제돼야 → 멱등성이 중요한 이유
🏗️ 자주 쓰는 구조와 패턴
옵저버 패턴 / Observer Pattern

어떤 값이 바뀌면 구독 중인 여러 곳에 자동으로 알려주는 구조. 이벤트 리스너, Firebase의 onSnapshot이 이 패턴.

낙관적 업데이트 / Optimistic Update

서버 응답을 기다리지 않고 UI를 먼저 바꾸는 것. 빠르게 느껴지지만 실패하면 롤백해야 함.

좋아요 버튼 누르면 즉시 하트가 채워지고, 서버 실패 시 되돌아오는 것이 낙관적 업데이트예요.
이벤트 버블링 / Event Bubbling

자식 요소의 이벤트가 부모로 전파되는 것. 카드 안의 버튼 클릭이 카드 클릭도 같이 발생시키는 이유.

// 버블링 막기 button.onclick = function(e) { e.stopPropagation(); // 부모로 전파 차단 doSomething(); };
모킹 / Mocking

실제 데이터나 API 대신 가짜로 만들어서 테스트하는 것. 서버가 아직 없어도 프론트 개발 가능하게 해줌.

AI가 "일단 목업 데이터로 개발하고 나중에 실제 API 연결할게요"라고 하면 이 뜻.
SSOT / Single Source of Truth

데이터의 유일한 원본을 한 곳에만 두는 원칙. 두 곳에 같은 데이터가 있으면 동기화 문제가 생김.

Modal · Dialog · Popup · Overlay자주 혼동

모두 "현재 화면 위에 떠있는 무언가"예요.

Modal (모달): 화면 중앙에 뜨는 창. 뒤 배경이 어두워지고, 닫기 전까지 다른 조작이 막혀요.
Overlay (오버레이): Modal을 띄울 때 배경에 깔리는 어두운 반투명 레이어.
Popup (팝업): 요즘은 Modal과 거의 같은 뜻으로 혼용돼요.

Toast · Snackbar · Tooltip · Banner

Toast (토스트): 화면 아래에 잠깐 나타났다 사라지는 짧은 알림. "저장됐어요 ✓" 같은 것.
Snackbar (스낵바): Toast와 거의 같아요. Android에서 쓰는 표현.
Tooltip (툴팁): 요소에 마우스를 올리면 나오는 작은 설명 풍선.
Banner (배너): 화면 상단에 길게 가로질러 표시되는 알림 바.

FAB · Chip · Badge · Breadcrumb

FAB (Floating Action Button): 화면 위에 둥둥 떠있는 큰 원형 버튼.
Chip (칩): 작고 둥근 태그 모양 요소. 필터, 태그, 선택지 등에 써요.
Badge (뱃지): 숫자나 점으로 알림 개수를 표시하는 작은 표시.
Breadcrumb (브레드크럼): 현재 위치를 계층으로 보여주는 탐색 표시. 홈 > 설정 > 디자인

Bottom Sheet · Drawer · Sidebar

Bottom Sheet (바텀 시트): 화면 아래에서 위로 밀려 올라오는 패널. 모바일에서 자주 써요.
Drawer (드로어): 평소엔 숨겨져 있다가 햄버거 버튼을 누르면 옆에서 밀려나오는 메뉴.
Sidebar (사이드바): 화면 옆(주로 왼쪽)에 항상 노출되는 보조 영역.

Dropdown · Popover · Context Menu

Dropdown (드롭다운): 버튼을 클릭하면 아래로 펼쳐지는 선택 목록.
Popover (포오버): 요소 근처에 말풍선처럼 뜨는 작은 패널. Tooltip보다 크고 더 많은 내용을 담아요.
Context Menu: 마우스 우클릭하면 나오는 메뉴.

✍️ 렌더링과 에디터 용어
렌더링(Rendering) — 코드를 눈에 보이는 화면으로자주 나옴

CSR (Client-Side Rendering): 브라우저가 JavaScript를 받아서 화면을 직접 그려요.

SSR (Server-Side Rendering): 서버가 HTML을 미리 만들어서 브라우저에 보내줘요. Next.js가 대표적.

SSG (Static Site Generation): 빌드할 때 HTML을 미리 다 만들어 저장해요. 속도 최강.

CSR: 브라우저 → JS 실행 → 화면 그리기 SSR: 서버 → HTML 완성 → 브라우저에 전송 SSG: 빌드 시 HTML 생성 → CDN에 저장 → 즉시 전송
WYSIWYG, Markdown, Hot Reload

WYSIWYG (위지윅): 편집 화면에서 보이는 그대로가 최종 결과물. 노션, 구글 독스가 대표적.

Markdown (마크다운): #, **, - 같은 기호로 서식을 표현하는 경량 문법. GitHub README에 많이 써요.

Hot Reload (핫 리로드): 코드를 저장하면 브라우저 전체를 새로고침하지 않고 변경된 부분만 자동 반영.

Template · Boilerplate · Scaffold

Template (템플릿): 미리 만들어둔 틀. 내용만 채우면 돼요.
Boilerplate (보일러플레이트): 프로젝트 시작 시 매번 필요한 반복 코드 묶음.
Scaffold (스캐폴드): 프레임워크가 자동으로 기본 파일·폴더 구조를 생성해주는 것.

Template = 빈칸 채우기 양식 📄 · Boilerplate = 항상 쓰는 기본 세팅 🔩 · Scaffold = 공사 시작 전 세우는 비계 🏗️
📝 용어 확인 퀴즈
얼마나 익혔나 확인해봐요

Q. 기능은 그대로인데 코드 구조를 깔끔하게 개선하는 것을 뭐라고 할까요?

리팩토링(Refactoring)은 결과는 동일하게 유지하면서 코드 품질을 개선하는 것이에요.

Q. API에서 특정 기능을 담당하는 URL 주소를 뭐라고 해요?

엔드포인트(Endpoint)예요. /api/users 같은 URL 경로 하나하나가 엔드포인트예요.

Q. 이전에 잘 되던 기능이 수정 후에 망가지는 것을 뭐라고 해요?

리그레션(Regression)이에요. 수정 전에 잘 되던 기능이 수정 후에 망가지는 것. 테스트가 중요한 이유예요.

Q. 코드 밖에 저장하는 API 키, 비밀번호 같은 설정값을 뭐라고 해요?

환경변수(Environment Variable)예요. .env 파일에 저장하고 절대 GitHub에 올리면 안 돼요!

Q. 먼저 넣은 것이 먼저 나오는 자료구조는?

큐(Queue)예요. FIFO(First In, First Out). 줄 서기랑 같아요. 스택은 반대로 LIFO(마지막에 넣은 게 먼저 나옴).

Q. 화면 아래에서 위로 밀려 올라오는 패널을 뭐라고 하나요?

Bottom Sheet예요. 모바일에서 공유·정렬·옵션 패널로 자주 써요.

Q. 노션처럼 편집 화면에서 보이는 그대로 최종 결과가 되는 방식은?

WYSIWYG(위지윅)이에요. What You See Is What You Get. 노션, 구글 독스가 대표적이에요.

Q. 데이터를 다루는 기본 4가지 동작(만들기·읽기·수정·삭제)을 묶어 부르는 말은?

CRUD예요. Create / Read / Update / Delete의 앞글자. 대부분의 앱이 이 네 가지 동작 중심으로 만들어져요.

Q. 잠깐 나타났다 사라지는 "저장됐어요 ✓" 같은 짧은 알림의 이름은?

Toast(또는 Snackbar)예요. 짧은 피드백 알림으로 자주 써요.
